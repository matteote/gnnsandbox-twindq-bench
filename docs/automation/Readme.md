# Network Automation

The lifecycle automation code that manages the linux networking to deploy a virtual network is provided as a GKE k8s operator (`operator/`). 

The Kubernetes operator is built on the [kopf](https://kopf.readthedocs.io/en/latest/) framework. It manages the full lifecycle of virtual network functions and lab infrastructure by watching Custom Resources and driving Ansible playbooks against the Network VM. 

Resources are reconciled declaratively — create, update and delete operations are all handled automatically.

## Operator Logs

To see the logs from the operator run the following commands

```bash
kubectl get pods
# NAME                          READY   STATUS    RESTARTS   AGE
# free5gc-operator-<UUID>        1/1     Running   0          9m22s

kubectl logs -f free5gc-operator-<UUID>
```

## Network Component Lifecycle Overview

| Component | Short name | Description |
|---|---|---|
| `VyOSInfrastructure` | `infra` | Declares the full network topology — routers, links, and locations. Generates child VyOSRouter and LinuxNetwork resources. |
| `VyOSUnderlay` | `underlay` | Configures underlay routing protocols (OSPF, BGP, MPLS/LDP) across routers defined in an infrastructure. Depends on VyOSInfrastructure. |
| `VyOSL3VPN` | `l3vpn` | Applies L3VPN VRF services and per-VRF BGP neighbours to PE routers. Depends on VyOSUnderlay. |
| `VyOSRouter` | `vr` | Represents a single VyOS router container. Usually generated by VyOSInfrastructure but can be created directly. |
| `LinuxNetwork` | `lnet` | Represents a Linux bridge / Docker network segment. Usually generated by VyOSInfrastructure. |
| `Device` | `dev` | A simulated end-user device or CPE container attached to a LinuxNetwork. |
| `TrafficTest` | `tt` | Runs a traffic test between one or more source Devices and a destination Device using the traffic-agent binary. |
| `VyosVM` | `vyosvm` | Provisions a GCP VM to act as the Network Automation host that runs VyOS containers. |

### Dependency Chain

```
VyosVM (automation host)
  └── VyOSInfrastructure
        ├── LinuxNetwork  (generated)
        └── VyOSRouter    (generated)
              └── VyOSUnderlay
                    └── VyOSL3VPN
                          └── Device
                                └── TrafficTest
```

The operator enforces these dependencies automatically. Resources that depend on a parent will wait (retrying every 10–30 seconds) until the parent reaches `Ready` state before proceeding.

## CRD Reference

### VyOSInfrastructure

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get infra` or `kubectl get vyosinfrastructures`

#### Summary

The top-level resource for defining a virtual network topology. A single `VyOSInfrastructure` describes all routers and network segments. When created, the operator:

1. Validates the topology for consistency.
2. Generates one `LinuxNetwork` resource per network segment.
3. Generates one `VyOSRouter` resource per router.
4. Updates the GCP Ops Agent Prometheus scrape configuration to include all new routers.

On deletion, the scrape configuration is cleared. On operator restart, Ops Agent config is re-synced automatically via a `resume` handler.

#### Key spec fields

| Field | Description |
|---|---|
| `spec.networks[]` | List of network segments. Each has `name`, `subnet`, optional `vlan`, `mtu`, `network_type` (`p2p`, `multi-access`, `management`, `loopback`), `bandwidth`, and a `connected_routers` list specifying which router interface connects to it. |
| `spec.routers[]` | List of routers. Each has `name`, `hostname`, `router_id`, `role` (P/PE/CE), `image`, optional `location` (lat/long/city/country/site), and `interfaces` mapping interface names to network segments. |
| `spec.qos` | QoS policy definitions (bandwidth, class, DSCP). |
| `spec.security.firewall` | Firewall policy definitions (rules, actions, protocols). |

#### Example

```yaml
apiVersion: google.dev/v1
kind: VyOSInfrastructure
metadata:
  name: l3vpn-hub-spoke-infra
  namespace: default
spec:
  description: "L3VPN Hub-and-Spoke Infrastructure"
  networks:
    - name: p1-p2
      subnet: "172.16.30.0/24"
      vlan: 301
      network_type: "p2p"
      connected_routers:
        - router_name: "p1"
          interface: "eth1"
          ip_address: "172.16.30.1"
        - router_name: "p2"
          interface: "eth1"
          ip_address: "172.16.30.2"
    - name: mgmt
      subnet: "192.168.122.0/24"
      gateway: "192.168.122.1"
      network_type: "management"
      connected_routers:
        - router_name: "p1"
          interface: "eth0"
          ip_address: "192.168.122.11"
  routers:
    - name: "p1"
      hostname: "p1"
      router_id: "10.0.0.3"
      role: "P"
      location:
        latitude: 51.5074
        longitude: -0.1278
        city: "London"
        country: "United Kingdom"
      interfaces:
        - name: "eth0"
          network: "mgmt"
        - name: "eth1"
          network: "p1-p2"
        - name: "lo"
          network: "loopbacks"
```

See `environment/telco-lab/l3vpn-hub-spoke.yaml` for a full 12-router topology example.

### VyOSUnderlay

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get underlay` or `kubectl get vyosunderlays`

#### Summary

Applies underlay routing protocol configuration (OSPF, BGP, MPLS/LDP) on top of an existing `VyOSInfrastructure`. The operator waits for the referenced infrastructure to reach `Ready` status, then patches each named `VyOSRouter` resource with the per-router protocol configuration.

This allows the topology definition (`VyOSInfrastructure`) to remain separate from the protocol configuration, making it easy to swap underlay designs without recreating routers.

#### Key spec fields

| Field | Description |
|---|---|
| `spec.infrastructureRef` | Name of the `VyOSInfrastructure` to build on top of. Must be `Ready` before this resource is processed. |
| `spec.routing.ospf` | Global OSPF settings: `router_id_source`, area definitions. |
| `spec.routing.bgp` | Global BGP settings: `as_number`, `router_id_source`, `route_reflectors` list. |
| `spec.mpls` | Global MPLS settings: `enabled`, LDP `router_id_interface` and `label_range`. |
| `spec.routers[]` | Per-router protocol overrides. Each entry has a `name` (matching a router in the infrastructure) and a `protocols` block supporting `ospf`, `bgp`, `mpls`, and `static` sections. |

#### Lifecycle phases

`Waiting` → `Processing` → `Ready` / `Error`

#### Example

```yaml
apiVersion: google.dev/v1
kind: VyOSUnderlay
metadata:
  name: l3vpn-hub-spoke-underlay
  namespace: default
spec:
  infrastructureRef: l3vpn-hub-spoke-infra
  routing:
    ospf:
      router_id_source: "loopback"
      areas:
        - area_id: "0.0.0.0"
          type: "backbone"
    bgp:
      as_number: 65001
      router_id_source: "loopback"
      route_reflectors: ["10.0.0.1", "10.0.0.2"]
  mpls:
    enabled: true
    ldp:
      router_id_interface: "loopback"
  routers:
    - name: "p1"
      protocols:
        ospf:
          router_id: "10.0.0.3"
          areas:
            - area: "0.0.0.0"
              type: "backbone"
        mpls:
          enabled: true
          ldp:
            router_id: "10.0.0.3"
            interfaces: ["eth1", "eth2", "eth3", "eth4", "eth5"]
    - name: "rr1"
      protocols:
        bgp:
          as_number: 65001
          router_id: "10.0.0.1"
          route_reflector: true
          neighbors:
            - peer: "10.0.0.7"
              remote_as: 65001
              route_reflector_client: true
```

### VyOSL3VPN

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get l3vpn` or `kubectl get vyosl3vpns`

#### Summary

Applies Layer 3 VPN (VRF) services to PE routers. The operator waits for the referenced `VyOSUnderlay` to reach `Ready` status, then for each router in `spec.routers`:

1. Patches the `VyOSRouter` with VRF definitions (name, table, RD, RT import/export).
2. Assigns named interfaces to VRFs using a JSON patch.
3. Patches BGP VRF neighbours for each VRF.

This is the third layer of the dependency chain:  
`VyOSInfrastructure` → `VyOSUnderlay` → `VyOSL3VPN`

#### Key spec fields

| Field | Description |
|---|---|
| `spec.underlayRef` | Name of the `VyOSUnderlay` to build on. Must be `Ready` before proceeding. |
| `spec.services[]` | Global L3VPN service definitions: `name`, `type`, `topology` (hub/spoke/any-to-any), `rd`, `rt_export`, `rt_import`. |
| `spec.routers[]` | Per-router VRF and BGP configuration. Each router entry has: `name`, `vrfs[]` (name, table, rd, rt_export, rt_import, interfaces list), `bgp.vrfs[]` (per-VRF neighbours). |

#### Example

```yaml
apiVersion: google.dev/v1
kind: VyOSL3VPN
metadata:
  name: l3vpn-blue-service
  namespace: default
spec:
  underlayRef: l3vpn-hub-spoke-underlay
  services:
    - name: "BLUE_SPOKE"
      type: "l3vpn"
      topology: "spoke"
    - name: "BLUE_HUB"
      type: "l3vpn"
      topology: "hub"
  routers:
    - name: "pe1"
      vrfs:
        - name: "BLUE_SPOKE"
          table: 200
          rd: "10.50.50.1:1011"
          rt_export: ["65035:1011"]
          rt_import: ["65035:1030"]
          interfaces: ["eth2"]
      bgp:
        vrfs:
          - name: "BLUE_SPOKE"
            neighbors:
              - peer: "10.50.50.2"
                remote_as: 65035
```

### VyOSRouter

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get vr` or `kubectl get vyosrouters`

#### Summary

Represents a single VyOS router container running on the Network VM. `VyOSRouter` resources are usually created automatically by `VyOSInfrastructure`, but can also be created directly for standalone routers.

The operator:
- Waits for all referenced `LinuxNetwork` resources to be `Ready`.
- Uses Ansible to create the VyOS Docker container on the Network VM.
- Applies the full VyOS configuration (interfaces, VLANs, VRFs, OSPF, BGP, MPLS, static routes, DHCP, NAT, QoS, firewall).
- Monitors the container every 120 seconds and updates status on state changes.
- Syncs router status to Cloud Spanner for graph and agent use.

#### Key spec fields

| Field | Description |
|---|---|
| `spec.hostname` | Router hostname. |
| `spec.router_id` | Router ID (IPv4 address). |
| `spec.image` | VyOS container image, default `vyos:1.5`. |
| `spec.source_network` | Name of the `VyOSInfrastructure` that generated this router. |
| `spec.role` | Router role: `P`, `PE`, or `CE`. |
| `spec.location` | Physical location metadata (city, country, latitude, longitude). |
| `spec.interfaces[]` | Interface definitions: `name`, `address` (CIDR), `mtu`, `vlan`, `vrf`, `linux_network` (which LinuxNetwork bridge to connect to). |
| `spec.vrfs[]` | VRF definitions: `name`, `table`, `rd`, `rt_export`, `rt_import`. |
| `spec.protocols.ospf` | OSPF config: `router_id`, `areas` (networks, interfaces, type), `passive_interfaces`. |
| `spec.protocols.bgp` | BGP config: `as_number`, `router_id`, `route_reflector`, `neighbors`, `vrfs`, `address_families`. |
| `spec.protocols.mpls` | MPLS/LDP config: `enabled`, `ldp.router_id`, `ldp.interfaces`. |
| `spec.protocols.static` | Static routes: `destination`, `next_hop`, `distance`, `vrf`. |
| `spec.services.dhcp[]` | DHCP server configs per network. |
| `spec.services.nat` | NAT source rules (masquerade or address). |
| `spec.qos` | QoS policy references. |
| `spec.firewall` | Firewall policy references. |

#### Status phases

`Pending` → `Creating` → `Configuring` → `Running` / `Failed`

The status also exposes `container_id`, `ip_address`, `interfaces[]` (with up/down state), and `protocols_status` (OSPF/BGP/MPLS neighbour counts).

### LinuxNetwork

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get lnet` or `kubectl get linuxnetworks`

#### Summary

Represents a Linux bridge or Docker network segment on the Network VM. `LinuxNetwork` resources are usually created by `VyOSInfrastructure` but can be created directly. The operator uses Ansible to create the underlying bridge/network and connect router veth interfaces to it.

For `management` type networks, the status also reports the host interface, interface IP, and default gateway.

The operator monitors bridge state every 120 seconds, updating status on changes and syncing to Spanner (SCD Type 2) when bridge or veth state changes.

#### Key spec fields

| Field | Description |
|---|---|
| `spec.name` | Bridge/network name (max 15 chars). |
| `spec.source_network` | Parent `VyOSInfrastructure` name. |
| `spec.network_type` | `p2p`, `multi-access`, `loopback`, or `management`. |
| `spec.bandwidth` | Bandwidth cap (e.g. `1gbit`, `100mbit`). |
| `spec.gateway` | Gateway IP (for management networks). |
| `spec.connected_routers[]` | Routers to connect: `router_name`, `interface`, `ip_address`, optional `mac_address`. |

#### Status phases

`Pending` → `Creating` → `Ready` / `Failed`

### Device

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get dev` or `kubectl get devices`

#### Summary

Provisions a simulated end-user device or customer premise equipment (CPE) as an Alpine Linux Docker container on the Network VM. Devices attach to a `LinuxNetwork` bridge. They come pre-installed with networking tools (ping, traceroute, tcpdump) and the **traffic-agent** binary, and are used as traffic endpoints in `TrafficTest` resources.

The operator waits for the referenced `LinuxNetwork` to be `Ready` before creating the container. Device names must be 8 characters or fewer.

#### Key spec fields

| Field | Description |
|---|---|
| `spec.network_name` | Name of the `LinuxNetwork` to attach to. Must be `Ready`. |
| `spec.ip_address` | Static IPv4 address for the device (must be within the network subnet). |
| `spec.gateway` | Default gateway (typically the router interface IP on that network). |
| `spec.vlan` | Optional 802.1Q VLAN ID, must match the network segment. |

#### Status phases

`Pending` → `Creating` → `Ready` / `Failed`

#### Example

```yaml
apiVersion: google.dev/v1
kind: Device
metadata:
  name: dev1
  namespace: default
spec:
  network_name: "lan-spoke1"
  ip_address: "10.100.1.10"
  gateway: "10.100.1.1"
```

### TrafficTest

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get tt` or `kubectl get traffictests`

#### Summary

Runs a traffic test between one or more source `Device` resources and a single destination `Device` using the **traffic-agent** Go binary embedded in every device container image. The operator:

1. Validates all source and destination `Device` resources are `Ready`.
2. Assigns each source device a unique port via the `PortAllocator` (auto or manual mode).
3. Uses Ansible to `docker exec` `traffic-agent serve` on the destination and `traffic-agent run` on each source.
4. Tracks per-source and aggregate metrics (throughput, latency, packet loss, jitter, active sessions, TCP retransmissions).

Supports six traffic patterns: `constant`, `periodic` (sine/square/sawtooth), `burst`, `poisson`, `multi_sine` (N superimposed sine waves with optional UTC wall-clock anchor), and `schedule` (piecewise-linear bandwidth profile driven by HH:MM waypoints).

See [`traffic-agent/README.md`](/traffic-agent/README.md) and [`environment/telco-lab/traffictests.md`](/environment/telco-lab/traffictests.md) for full details.

#### Key spec fields

| Field | Description |
|---|---|
| `spec.source_devices[]` | List of source `Device` names. All must be `Ready`. |
| `spec.destination_device` | Destination `Device` name. Must be `Ready`. |
| `spec.protocol` | `TCP` or `UDP`. |
| `spec.port` | Base destination port (each source gets port + index). |
| `spec.duration` | Test duration in seconds. |
| `spec.bandwidth` | Target bandwidth e.g. `100Mbps`, `1Gbps`. |
| `spec.pattern_type` | Traffic pattern: `constant`, `periodic`, `burst`, `poisson`, `multi_sine`, or `schedule`. |
| `spec.pattern_config` | Pattern-specific settings (wave type, period, amplitude, burst rate/duration/interval, Poisson arrival rate, etc.). |
| `spec.concurrent_users` | Maximum concurrent sessions (default 1). |
| `spec.metrics_enabled` | Enable metrics collection (default false). |
| `spec.metrics_interval` | Metrics sampling interval in seconds (default 5). |

#### Status phases

`Pending` → `Deploying` → `Running` → `Completed` / `Failed` / `Stopped`

Status exposes per-source metrics in `source_status` and aggregate totals in `aggregate_metrics` (total throughput, avg latency, avg packet loss, total connections).

#### Example

```yaml
apiVersion: google.dev/v1
kind: TrafficTest
metadata:
  name: spoke-to-hub-test
  namespace: default
spec:
  source_devices:
    - dev1
    - dev2
  destination_device: devhub
  protocol: TCP
  port: 5201
  duration: 300
  bandwidth: "100Mbps"
  pattern_type: constant
  metrics_enabled: true
  metrics_interval: 5
```

### VyosVM

**Group/Version:** `google.dev/v1`  
**kubectl:** `kubectl get vyosvm`

#### Summary

Provisions the GCP Compute Engine VM that acts as the Network Automation host. This VM runs all VyOS router containers as Docker containers and is the target of all Ansible playbooks from the operator. The operator:

1. Allocates an external IP address.
2. Creates an `e2-standard-16` Ubuntu 22.04 GCP VM with the external IP attached.
3. Runs the VyOS VM install playbook to configure Docker, install FRR exporters and the Ops Agent.

This is typically a singleton resource created once during initial environment setup (`install.sh`).


## Common Operations

```bash
# List all network infrastructure resources
kubectl get infra,underlay,l3vpn,vr,lnet,dev,tt

# Check the status of all routers
kubectl get vr

# Watch traffic test progress
kubectl get tt -w

# Describe a failing resource for error details
kubectl describe vr p1
kubectl describe infra l3vpn-hub-spoke-infra

# Delete a full topology (child resources are garbage-collected)
kubectl delete infra l3vpn-hub-spoke-infra

# Deploy the full L3VPN hub-spoke lab topology
kubectl apply -f environment/telco-lab/l3vpn-hub-spoke.yaml
```
