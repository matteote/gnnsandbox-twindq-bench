# TrafficTest Resource

The TrafficTest resource enables realistic traffic simulation between Device containers in the virtual network for performance analysis, SLA validation, and GNN training data generation.

## Overview

The TrafficTest resource generates configurable traffic patterns that simulate real-world network usage scenarios. It supports six traffic pattern types, concurrent session simulation, and real-time metrics collection. Traffic is generated by the **traffic-agent** binary — a single static Go binary baked into every device container image — replacing the previous iperf3 + Python script approach.

## Features

- **Six Traffic Patterns**: Constant, periodic (sine/square/sawtooth), burst, Poisson, multi-sine (wall-clock aware), and schedule (waypoint-driven)
- **Concurrent Session Simulation**: Multiple simultaneous TCP/UDP sessions with configurable session durations and think times
- **Wall-Clock Time Reference**: `multi_sine` and `schedule` patterns can anchor to UTC time so patterns repeat consistently across long-running tests and multiple devices
- **Gaussian Noise**: `multi_sine` adds realistic stochastic variation on top of the bandwidth envelope
- **Real-time Monitoring**: Background monitoring with automatic status updates and test completion handling
- **Container-based Testing**: Works with Docker containers representing network devices; no external tools needed
- **Zero-dependency Binary**: `traffic-agent` is statically compiled — no Python, no iperf3 required in the container

## Architecture

```
TrafficTest CR → Operator → Ansible → networkvm
                     ↓           ↓
               Background    docker exec → device container
               Monitoring           └── traffic-agent serve  (destination)
                     ↓               └── traffic-agent run   (source)
               Status Updates & Completion
```

The system uses:
- **Kubernetes Operator** (`operator/src/traffictest/`): Manages TrafficTest custom resources via kopf
- **Ansible Playbooks** (`operator/src/traffictest/playbooks/`): Orchestrate `docker exec` calls against device containers on the networkvm
- **traffic-agent** (`traffic-agent/`): Go binary embedded in device images — handles both traffic generation (source) and reception (destination)
- **Background Monitoring**: Async coroutine in the operator tracks test completion and updates Kubernetes status

See [`traffic-agent/README.md`](/traffic-agent/README.md) for full details on the binary.

## Traffic Patterns

### 1. Constant Pattern
Maintains steady bandwidth throughout the test duration.

```yaml
spec:
  pattern_type: constant
  bandwidth: 50Mbps
  concurrent_users: 10
```

### 2. Periodic Pattern
Varies bandwidth over time using a single mathematical wave.

```yaml
spec:
  pattern_type: periodic
  pattern_config:
    wave_type: sine      # sine, square, or sawtooth
    period: 3600         # seconds (1 hour cycle)
    base_rate: 20Mbps
    amplitude: 30Mbps    # varies between base_rate ± amplitude
```

### 3. Burst Pattern
Alternates between high-traffic bursts and low-traffic idle periods.

```yaml
spec:
  pattern_type: burst
  pattern_config:
    burst_duration: 180    # 3 minutes of high traffic
    burst_interval: 600    # every 10 minutes
    burst_rate: 100Mbps    # traffic during bursts
    idle_rate: 5Mbps       # traffic during idle periods
```

### 4. Poisson Pattern
Simulates realistic user arrivals using Poisson distribution.

```yaml
spec:
  pattern_type: poisson
  pattern_config:
    arrival_rate: 2.0      # users per second
  concurrent_users: 100    # maximum concurrent
  session_duration: 300    # average session length
```

### 5. Multi-Sine Pattern
Superposes N independent sine waves on a base rate. Supports `wall_clock` time reference so patterns peak at the same UTC hour every day, regardless of test start time. Ideal for GNN training data generation.

```yaml
spec:
  pattern_type: multi_sine
  bandwidth: 50Mbps          # base_rate
  pattern_config:
    base_rate: 50Mbps
    time_reference: wall_clock   # anchor to UTC epoch
    noise_stddev_pct: 3.0        # ±3% Gaussian noise
    min_rate: 5Mbps
    max_rate: 95Mbps
    components:
      - period: 86400            # 24-hour daily cycle
        amplitude: 30Mbps
        phase_offset: -28800     # peak at 14:00 UTC
      - period: 604800           # 7-day weekly cycle
        amplitude: 10Mbps
        phase_offset: 0
```

### 6. Schedule Pattern
Piecewise-linear (or step) bandwidth profile driven by named UTC waypoints. Repeats daily, weekly, or once.

```yaml
spec:
  pattern_type: schedule
  pattern_config:
    time_reference: wall_clock
    interpolation: linear     # or "step"
    repeat: daily
    waypoints:
      - time: "00:00"  rate: "5Mbps"
      - time: "09:00"  rate: "80Mbps"
      - time: "12:30"  rate: "60Mbps"
      - time: "14:00"  rate: "90Mbps"
      - time: "18:00"  rate: "30Mbps"
      - time: "22:00"  rate: "10Mbps"
```

## Configuration Reference

### Required Fields

- `source_devices`: List of source Device names (must be an array, even for a single device)
- `destination_device`: Name of the destination Device
- `protocol`: TCP or UDP

### Optional Fields

| Field | Default | Description |
|-------|---------|-------------|
| `port_mode` | `auto` | `auto` (operator picks free port) or `manual` (use `port` field) |
| `port` | — | Manual port number (required if `port_mode: manual`) |
| `duration` | 60 | Test duration in seconds |
| `bandwidth` | 10Mbps | Bandwidth target (constant or base rate) |
| `pattern_type` | `constant` | Traffic pattern type |
| `pattern_config` | `{}` | Pattern-specific settings |
| `concurrent_users` | 1 | Maximum concurrent sessions |
| `session_duration` | — | Average session duration (seconds, Poisson) |
| `think_time` | 0 | Pause between requests (seconds) |
| `metrics_enabled` | false | Enable metrics collection |
| `metrics_interval` | 5 | Metrics sampling interval (seconds) |

```yaml

  name: basic-test

  source_devices:
    - customer-a-device-1
  destination_device: customer-b-device-1

  duration: 300
  bandwidth: 50Mbps
  concurrent_users: 10
```

```json
  "source_device":   "dev1",

  "end_time":        "2025-10-30T10:05:00Z",
  "duration_sec":    300,
  "pattern_type":    "constant",
  "completed":       true,
  "returncode":      0,
  "metrics": {
    "bytes_sent":      18750000000,
```

### Log Files and Locations (inside device containers)

| File | Description |
|------|-------------|
| `/tmp/traffic_test_<name>_<port>/config_<port>.json` | JSON config written by Ansible |
| `/tmp/traffic_test_<name>_<port>/traffic_test.log` | traffic-agent stdout/stderr |
| `/tmp/traffic_test_<name>_<port>/results.json` | Final results JSON |
| `/tmp/traffic_test_<name>_<port>/exit_code` | Process exit code |

## Troubleshooting

### Common Issues

1. **Device Not Ready**: Ensure source and destination Devices are in `Ready` phase before creating a TrafficTest
2. **Port Conflicts**: Use `port_mode: auto` (the default) — the operator allocates conflict-free ports automatically
3. **Agent Not Found**: If `traffic-agent` binary is missing, the device image was built before the agent was added — delete and re-create the Device to trigger a fresh container start
4. **Process Not Running**: Check container logs; a non-zero exit code is written to the `exit_code` file
5. **Network Connectivity**: Verify containers can reach each other on the allocated port

### Performance Considerations

#### Resource Usage
- Each concurrent session is a goroutine — very lightweight compared to the old per-session iperf3 process model
- Memory usage is low: a single static binary handles all sessions
- CPU usage scales with bandwidth and session count but remains much lower than spawning multiple processes

#### Scaling Guidelines
- **Small tests**: 1–10 concurrent sessions, < 100 Mbps
- **Medium tests**: 10–50 concurrent sessions, 100 Mbps–1 Gbps
- **Large tests**: 50+ concurrent sessions, > 1 Gbps

## Implementation Details

### Traffic Generation Process

1. **Operator receives TrafficTest CR**: Validates source/destination Devices are `Ready`
2. **Port allocation**: Operator assigns a unique port to each source (auto or manual)
3. **Ansible playbook execution** (`traffic_create.yaml`): Runs against the networkvm
4. **Destination setup**: `docker exec -d <dest> traffic-agent serve --port P --protocol TCP --duration N+30`
5. **Source setup**: Config JSON is `docker cp`'d into the source container; `docker exec -d <source> traffic-agent run --config /tmp/.../config_P.json`
6. **Background monitoring**: Async coroutine in the operator polls status every `metrics_interval` seconds and updates the Kubernetes status subresource

### File Structure

```
traffic-agent/                     # Go binary source
operator/src/traffictest/
├── lifecycle.py                   # kopf operator handlers (create/delete/monitor)
├── lifecycle_tasks.py             # Async Ansible execution helpers
├── port_allocator.py              # Thread-safe port allocation
└── playbooks/
    ├── traffic_create.yaml        # Start source + destination agents
    ├── traffic_delete.yaml        # Stop and clean up
    └── traffic_status.yaml        # Query agent status

operator/src/vyosvm/playbooks/roles/device/
├── files/Dockerfile               # Device image (pulls agent binary from Artifact Registry)
├── files/entrypoint.sh            # Container entrypoint
├── defaults/main.yaml             # agent_image and agent_version variables
└── tasks/main.yaml                # Build device image with --build-arg AGENT_IMAGE=...
```

### Container Requirements

Device containers need only:
- **traffic-agent binary** at `/usr/local/bin/traffic-agent` (baked in at image build time)
- **Network connectivity** between source and destination containers
- **Shell** (`/bin/sh`) for the Ansible `docker exec` wrapper

No Python, no iperf3, no extra packages.

## Contributing

To extend the TrafficTest resource:

1. **New Traffic Patterns**: Add a new file under `traffic-agent/internal/patterns/` implementing the `Pattern` interface, then register it in `factory.go`. Add the new enum value to `traffic.yaml` CRD.
2. **New Metrics**: Extend `traffic-agent/internal/metrics/collector.go` and add fields to the CRD `status.current_metrics` schema.
3. **Daemon Mode (Phase 2)**: Change `entrypoint.sh` to `exec /usr/local/bin/traffic-agent daemon --control-port 9090` and switch `lifecycle_tasks.py` from Ansible to direct HTTP calls.
4. **Documentation**: Update this file and `traffic-agent/README.md`.
